# プッシュ前の CI 検証ルール

**プッシュ前に CI で実行される全チェックをローカルで通すこと。これは必須であり、省略してはならない。**

---

## 必須手順

プッシュする **前に**、以下の手順を **毎回** 実行する。

### Step 1: CI ワークフローの読み取り

`.github/workflows/` 内の全ワークフローファイルを読み、`push` または `pull_request` トリガーで実行される `run:` ステップのコマンドを **すべてリストアップ** する。

### Step 2: ローカル実行可能なコマンドを特定

リストアップしたコマンドのうち、ローカルで実行可能なものを特定する。

**スキップしてよいもの（これ以外はスキップ不可）:**

- CI 固有のシークレットや環境変数が必要なコマンド（Sentry upload 等）
- GitHub API 操作（PR コメント投稿、ラベル付与等）
- Playwright 等の E2E テスト（明示的に求められない限り）
- Docker や特殊な CI 環境でのみ動作するコマンド

### Step 3: 全コマンドを実行して pass を確認

特定した全コマンドを実行する。**1 つでも fail したら修正してから再実行する。**

典型的なチェック対象（ワークフローに含まれている場合）:

- lint（`pnpm lint` 等）
- test（`pnpm test` 等）
- 未使用コード検出（`pnpm knip` 等）
- changeset チェック（changeset ファイルの有無）
- ライセンスチェック
- ビルド

### Step 4: プロジェクト固有のチェック

ワークフローにプロジェクト固有のチェック（changeset、ライセンス検証等）がある場合はそれも実行する。

例: changeset チェックがある場合、コード変更を含む PR には changeset ファイルが必要。`.changeset/` に新規ファイルがなければ作成する。

### Step 5: 全 pass を確認してからプッシュ

---

## 禁止事項

- CI チェックを通さずにプッシュしてはならない
- lint エラーを disable コメントで安易に抑制してはならない（正当な理由がある場合は理由を明記）
- テスト失敗を無視してプッシュしてはならない
- **「たぶん大丈夫」でプッシュしてはならない。実行して確認すること**
